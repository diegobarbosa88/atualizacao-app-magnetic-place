# Plano: Fase 6 - Infraestrutura de Estado e Utilitários

Este plano foca na preparação da base de código para a modularização, centralizando o estado e isolando funções utilitárias.

## 🎯 Objetivos
- Criar sistema de gerenciamento de estado global (Context API).
- Isolar funções auxiliares de data, hora e IA em módulos dedicados.
- Reduzir a carga de lógica de dados no componente `App`.

## 🛠️ Tarefas

### 1. Criar Módulos de Utilitários
- [ ] Criar `src/utils/dateUtils.js` e mover:
  - `toISODateLocal`, `isSameMonth`, `getLastBusinessDayOfMonth`, `formatDocDate`.
- [ ] Criar `src/utils/formatUtils.js` e mover:
  - `formatHours`, `calculateDuration`, `calculateExpectedMonthlyHours`, `getScheduleForDay`.
- [ ] Criar `src/utils/aiUtils.js` e mover:
  - `callGemini`.

### 2. Criar Contexto Global
- [ ] Criar `src/context/AppContext.jsx`.
- [ ] Mover configuração do Supabase e variáveis `supabaseUrl`/`supabaseKey`.
- [ ] Mover todos os estados principais (`workers`, `clients`, `logs`, etc.).
- [ ] Mover lógica de subscrição Realtime.
- [ ] Mover funções `saveToDb` e `handleDelete`.

### 3. Integração Inicial
- [ ] Modificar `main.jsx` (ou onde o App é renderizado) para incluir o `AppProvider`.
- [ ] Começar a consumir o Contexto no `app.jsx`.

## 🧪 Verificação
- [ ] Verificar se as datas e cálculos de horas continuam corretos.
- [ ] Confirmar se as atualizações em tempo real (Supabase) continuam funcionando após a migração para o Contexto.
- [ ] Garantir que o login (admin/worker) mantém o estado persistente no Contexto.

## ⚠️ Riscos
- **Re-renders**: O Contexto pode causar re-renders excessivos se não for otimizado com `useMemo`.
- **Dependências Circulares**: Cuidado ao mover funções que dependem de variáveis de estado.
