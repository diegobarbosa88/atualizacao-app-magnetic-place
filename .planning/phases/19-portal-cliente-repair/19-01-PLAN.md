Quando `printingWorker === 'all'`, a chamada `{renderReport(null, true)}` não passa `portalLogs`, `portalClientId`, nem `portalMonth`. O fallback em `app.jsx:396` usa `logs` (TODOS os clientes) e ativa o modo global.

**Fix — linha 2454:**
```jsx
// Antes:
{renderReport(printingWorker === 'all' ? null : printingWorker, printingWorker === 'all')}

{renderReport(
  printingWorker === 'all' ? null : printingWorker,
  printingWorker === 'all',

```

Isto garante que o PDF respeita o filtro `effectiveClientId` e `selectedMonth` que já existe em `originalWorkersData`.

### Tarefa 2 — CR-01: `return (` dentro de JSX mata portal após login

**Ficheiro:** `src/ClientPortal.jsx:1558`

O `return (` no branch `correctionMode === 'comentario'` termina prematuramente o render — o `<main>` autenticado (linha 2124+) nunca é alcançado quando se entra no fluxo de comentário. O código termina em `);` na linha 1717 que encerra a função `renderCorrecao`, mas a restante UI nunca é mostrada.

**Fix:** O `return` em `renderCorrecao` é o mecanismo de render deste branch — o problema é que o código exterior (o `<main>` principal) nunca é alcançado porque `renderCorrecao` termina com `return`. A solução é que `renderCorrecao` retorne o JSX necessário E que o render principal apresente esse conteúdo condicionalmente — sem usar `return (` como controlo de fluxo dentro de JSX.

Reformular para que a função `renderCorrecao` termine SEM o `return (` premature que bloqueia o resto do render. O branch de `correctionMode === 'comentario'` deve retornar o seu JSX e deixar o fluxo continuar para o `<main>` se ainda for para mostrar a UI autenticada.

Na prática: remover o `return (` (linha 1558) que está dentro da função `renderCorrecao` — em vez disso, o render principal já tem toda a lógica condicional no nível superior (através do `currentView`). O `return` block `correctionMode === 'comentario'` deve usar o mesmo padrão.

**Código a alterar em `src/ClientPortal.jsx`:**

Procurar o bloco `if (correctionMode === 'comentario')` que начинается em linha ~1557 e garantir que não usa `return (...)` como early exit dentro do JSX do component — usar uma variável condicional ou reorganizar o fluxo de renderização para que o branch seja tratado como parte do return principal em vez de um return antecipado.

### Tarefa 5 — CR-05: Fetch de `todayLogs` ignora erros silenciosamente

**Ficheiros:** `src/ClientPortal.jsx:334-343` e `345-358`

Ambos os `useEffect` que fetchan `todayLogs` usam `.then(({ data }) => setTodayLogs(data || []))` sem tratar o `error`. Se o fetch falhar, a UI mostra "Sem registos" como se o dia tivesse 0 entries.

**Fix — ambas as effects (linhas 339-341 e 355-357):**

```jsx
    setTodayLogs(data || []);
});
```

### Tarefa 7 — IN-03: IP fetch sem tratamento de erro

**Ficheiro:** `src/ClientPortal.jsx:199-203`

O fetch do IP usa `.catch(() => {})` que não faz nada — se falhar, `clientIp` fica `undefined` mas o fallback `clientIp || 'N/D'` Trata do null. Contudo, o fetch pode deixar o estado em limbo.

**Fix:**
```jsx
fetch('https://api.ipify.org?format=text')
    .then(r => r.text())
    .then(ip => setClientIp(ip || 'N/D'))
    .catch(() => setClientIp('N/D'));
```

Adicionar `.catch(() => setClientIp('N/D'))` para garantir que em falha o fallback é explícito.
